## Problem Statement

F1 fans (especially the millions who came from Drive to Survive) watch races without understanding what's actually happening strategically. They hear "Piastri is on a one-stop" and have no idea what that means or why it matters. Current solutions either show raw data for power users (MultiViewer) or do nothing to explain the race.

The core pain: watching F1 is exciting, but understanding F1 requires knowledge that takes years to accumulate. No product bridges that gap in real time during a live race.

## What Makes This Cool

A real-time AI co-commentator that sits alongside the F1 TV broadcast. Not a chatbot. Not a sidebar full of numbers. A voice that explains what just happened, why it matters, and what's about to happen next. It knows when to talk and when to shut up.

"Race engineer in your ear, but only when you need one."

The voice coexists with the real broadcast commentary. Users toggle between Voice, Cards, or Both depending on whether they're listening to Sky/ESPN or watching on mute.

## Constraints

- Phase 1 is extension-only (no standalone web app). Web app is a Phase 2 consideration.
- Must work as a browser extension on F1 TV (Plasmo framework)
- OpenF1 API for live timing data (free, community-maintained)
- F1 TV stream has variable delay (need sync mechanism)
- AI explanations must be fast enough for live racing (~1-3s latency acceptable)
- Cannot depend on official F1 licensing or partnerships
- Side project — one developer, building for personal use and sharing

## Premises

1. AI explanation for casual fans is the differentiator, not data density. MultiViewer already wins on raw data. We win on understanding.
2. Browser extension on F1 TV is the right first form factor — fans are already there.
3. Voice commentary is the "whoa" moment and the biggest market gap. No one does this for F1.
4. Core wedge = Drive to Survive fans who don't understand F1 strategy yet.

## Approaches Considered

### Approach A: Weekend Warrior (Minimal)
Plasmo extension with sidebar + voice toggle. 6 event types, template-based explanations, browser TTS, replay mode. Ships in a weekend with CC. No auth, no social.

### Approach B: Race Engineer (Chosen — phased)
Full extension with event engine → explanation engine → voice orchestration pipeline. Phase 1: Beginner mode only, chrome.storage.sync, template + async LLM enrichment. Phase 2: Supabase auth, all 3 fan modes (Beginner / Regular / Advanced), friends/social.

### Approach C: Pure Voice (Lateral)
Audio-only extension — zero UI sidebar. Just a voice. Radically simple but no visual fallback.

## Recommended Approach

**Approach B: Race Engineer** — the full pipeline architecture.

Rationale: This is a side project you'll actually use during races. Building the minimal version means you'll want to rebuild it almost immediately. The full pipeline (event engine → explanation engine → voice orchestration) is the correct architecture regardless of scope — might as well build it right.

The key risk is LLM latency during live races. Mitigation: template-based fast path for common events (pit stops, safety car), LLM rewrite for uncommon events and for making templates sound natural.

### Architecture

```
OpenF1 API (WebSocket)
    │
    ▼
Event Engine (filters noise → race moments)
    │
    ├── Pit stop detected
    ├── Safety car / VSC
    ├── Position change (overtake)
    ├── Tyre degradation signal
    ├── DRS activation zone
    └── Gap collapse / undercut window
    │
    ▼
Explanation Engine (moment → plain English)
    │
    ├── Template path (fast, <100ms)
    │   "Piastri pits from P3. Switching from medium to hard tyres.
    │    He's covering Hamilton's undercut threat."
    │
    └── LLM path (richer, 1-3s)
        "Piastri just made a bold call — pitting before Hamilton to
         force the undercut. If his out-lap is fast enough, he'll
         jump Hamilton in the pit window."
    │
    ▼
Voice Orchestration
    │
    ├── Cards mode → render in sidebar
    ├── Voice mode → browser TTS or ElevenLabs
    └── Both mode → card + audio simultaneously
    │
    ├── Quiet detector: if TV broadcast audio is explaining
    │   the same moment, skip voice (V2 feature)
    └── Queue manager: don't stack 3 events, prioritize
```

### Tech Stack

- **Extension:** Plasmo (React-based Chrome/Firefox extension framework)
- **Live data:** OpenF1 API (WebSocket for live, REST for historical)
- **Explanation engine:** Claude API (haiku for speed) with template fallback
- **Voice:** Browser SpeechSynthesis API (Phase 1), ElevenLabs (Phase 1 upgrade if TTS quality fails test)
- **Backend:** None in Phase 1 (chrome.storage.sync). Supabase in Phase 2 (auth, preferences, friend rooms)
- **State management:** Zustand (lightweight, works well in extensions)
- **Stream sync:** Manual slider (-30s to +120s), auto-detect later

### Event Detection Spec

Each event type maps to specific OpenF1 API endpoints and trigger conditions:

| Event Type | OpenF1 Source | Trigger Condition |
|-----------|--------------|-------------------|
| Pit stop | `pit` endpoint (WebSocket) | `pit_duration` appears for a driver AND `stints` shows compound change |
| Safety car / VSC | `race_control` endpoint | Message contains "SAFETY CAR" or "VIRTUAL SAFETY CAR" |
| Overtake | `position` endpoint | Driver's position number decreases between consecutive updates |
| Tyre degradation | `stints` endpoint + `car_data` | Stint length exceeds expected window for compound (soft >20 laps, medium >35, hard >45) |
| DRS activation | `car_data` endpoint | `drs` field changes to active AND gap to car ahead <1.0s |
| Gap collapse / undercut | `intervals` endpoint | Gap between consecutive drivers decreases by >0.5s in 3 consecutive updates |

### Template/LLM Routing Rules

- **All 6 core event types use templates by default** (instant, <100ms)
- LLM (Claude Haiku) is called async to enrich the template text
- **If LLM response arrives within 2 seconds:** replace template card with enriched version
- **If LLM response takes >2 seconds:** template stands as-is
- **Latency gate:** Before shipping, benchmark Claude Haiku p95 latency for 2-sentence generation. If p95 >2s consistently, ship Phase 1 as template-only with LLM enrichment deferred to Phase 2.

### Stream Sync Spec

The sync offset applies to **event card/voice rendering delay**, not to leaderboard data.

- Leaderboard updates in real-time (always current)
- Event cards and voice commentary are delayed by the sync offset
- Default offset: 0s (user adjusts manually)
- Range: -30s to +120s (negative = user's stream is ahead of OpenF1, rare but possible)
- Stored in `chrome.storage.sync` so it persists across sessions
- V2: auto-detect offset by correlating OpenF1 events with audio analysis of the stream

### Voice Quality Risk

**Risk:** Browser `SpeechSynthesis` API voices are widely regarded as robotic. If the voice sounds bad, users will mute it within minutes, destroying the core "whoa" moment (success criterion #3).

**Mitigation strategy:**
- Phase 1: Ship with browser TTS as default, but include a "Voice Quality" setting
- If the builder (you) mutes it after testing, immediately switch to ElevenLabs for Phase 1
- ElevenLabs free tier: 10,000 characters/month. A typical race generates ~50-100 event explanations at ~100 chars each = ~5,000-10,000 chars. Tight but feasible for 1 race/month on free tier.
- **Decision point:** After first live race test, decide whether to upgrade voice to ElevenLabs in Phase 1 or defer.

### Event Voice Priority Queue

When multiple events fire within 10 seconds, voice follows this priority:

1. **Safety car / VSC** (highest — changes entire race)
2. **Pit stop** (strategy-defining)
3. **Overtake for position** (exciting, time-sensitive)
4. **Undercut / gap collapse** (strategic, can wait 5s)
5. **DRS activation** (contextual, lowest priority)
6. **Tyre degradation** (advisory, can queue)

**Queue rules:**
- Max 2 events in voice queue at any time
- If queue is full, lower-priority event gets card-only (no voice)
- Voice events auto-dismiss after 8-12 seconds (matching design review decision)

### OpenF1 Reliability & Fallback

**Risk:** OpenF1 is community-maintained with no SLA. If it goes down mid-race, the product is non-functional.

**Phase 1 fallback:**
- Cache the last 30 seconds of timing data locally
- If WebSocket disconnects: show "Live data paused" banner (from design review), display cached leaderboard with staleness indicator
- Attempt reconnect every 5 seconds with exponential backoff (max 30s)
- Event engine pauses (no phantom events from stale data)

**Phase 2 fallback:**
- Secondary data source evaluation (F1 official SignalR stream as backup)
- Record full race sessions to JSON for replay mode (also serves as offline fallback)

### Extension Lifecycle Error States

| State | What happens | User sees |
|-------|-------------|-----------|
| F1 TV page navigated away | Extension detects URL change | "Navigate to F1 TV to use F1 Companion" |
| Extension disabled/re-enabled | Sidebar resets, reconnects to OpenF1 | Brief loading state, auto-recovery |
| WebSocket disconnects | Reconnect with backoff | "Live data paused — reconnecting..." banner |
| F1 TV page layout changes | Extension detects missing mount point | "F1 TV layout changed — sidebar may not display correctly. Report issue." |
| No race active | OpenF1 returns no session data | "No live session. Try Replay Mode to demo." |

## Resolved Questions

1. **LLM latency:** Template-first with async LLM enrichment. Latency gate defined: if p95 >2s, ship template-only.
2. **OpenF1 reliability:** Cache-based fallback with degraded mode. Phase 2 adds secondary data source.
3. **Stream sync:** Manual slider applies to event card/voice rendering delay only, not to leaderboard data.
4. **Voice queue:** Priority ranking defined (SC > pit > overtake > undercut > DRS > tyre).
