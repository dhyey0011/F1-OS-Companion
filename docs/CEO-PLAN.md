## Vision

### 10x Check
Instead of a generic explainer, build an AI Race Engineer that feels like having a friend who's watched F1 for 20 years sitting next to you. It knows when to talk and when to shut up. It builds narrative tension ("Watch Hamilton's gap... it's been closing for 3 laps..."). Pre-race briefings set the stage. Post-race learning cards make fans smarter every weekend. The voice has personality, not just information. Replay mode becomes "learn F1 by re-watching legendary races with an AI guide."

### Platonic Ideal
The interface disappears. You're not reading cards or listening to a robot. You're experiencing a race with a companion that makes you smarter every weekend. The user feels: "I finally get F1. And it's fun." Every fan, from DTS newcomer to 20-year veteran, finds their depth level and never wants to watch without it again.

## Data Reliability

OpenF1 is community-maintained with no SLA. Fallback strategy:
- Cache last 30 seconds of timing data locally
- If polling fails 3 consecutive times: show "Live data paused" banner, display cached leaderboard with staleness indicator
- Retry every 5 seconds with exponential backoff (max 30s)
- Event engine pauses when data is stale (no phantom events)
- Rate limit: respect OpenF1's rate limits; if 429 received, back off exponentially

## Week 1 Feasibility Gate

These must be validated before building features:

1. **F1 TV DOM injection.** Validate that the Plasmo content script can inject a React root into a persistent sidebar div alongside F1 TV. If F1 TV's CSP blocks it, pivot to popup/side-panel extension.

2. **Polling reliability under tab switching.** Validate that the extension's polling loop continues when the F1 TV tab is backgrounded. If Chrome throttles timers, move polling to a service worker and relay data to the content script via `chrome.runtime.sendMessage`.

3. **Voice autoplay gate.** Validate that the Voice toggle click counts as a user gesture to unlock SpeechSynthesis. Test on Chrome AND Firefox.

## Feature Specifications

### 6 Core Event Types (from /office-hours design doc)

| Event Type | OpenF1 Source | Trigger Condition |
|-----------|--------------|-------------------|
| Pit stop | `pit` endpoint | `pit_duration` appears AND `stints` shows compound change |
| Safety car / VSC | `race_control` endpoint | Message contains "SAFETY CAR" or "VIRTUAL SAFETY CAR" |
| Overtake | `position` endpoint | Driver position decreases between consecutive updates |
| Tyre degradation | `stints` + `car_data` | Stint exceeds expected window (soft >18, medium >30, hard >40 laps) |
| DRS activation | `car_data` endpoint | `drs` field changes to active AND gap <1.0s |
| Gap collapse / undercut | `intervals` endpoint | Gap decreases by >0.5s in 3 consecutive updates |

### Template System

Templates are static string interpolations used as instant display while async LLM enrichment resolves.

Each event type has TWO variants (beginner + regular):

```
Pit stop:
  beginner: "{driver} just stopped for new tyres. They switched from {old} to {new}."
  regular:  "{driver} pits from P{pos}, {old} to {new}. Stint was {stint_laps} laps, {delta} vs expected window."

Safety car:
  beginner: "Safety car is out! All drivers slow down while the track is cleared."
  regular:  "SC deployed. {leader} has a {gap}s lead. Pit window opens for drivers on {compound}."
```

LLM enrichment must complete within 2s. If exceeded, template-only version is shown permanently for that event.

### Pre-race Briefing
- **Trigger:** 10 minutes before session `date_start` from OpenF1 `/sessions` endpoint. Race sessions only.
- **Content:** LLM prompt seeded with circuit name, grid positions, driver data from OpenF1
- **Output:** 3 storylines, each 1-2 sentences, rendered as event cards
- **Open issue:** OpenF1 does not expose championship standings. Need secondary source (Ergast API) or scope to grid positions + circuit info only.

### "What just happened?" Catch-up
- **Trigger:** User taps catch-up button (available when event feed has 3+ events)
- **Content:** LLM summarizes the last N events (N = events since last user tap/click, max 10). Default: last 5 events.
- **Output:** 2-3 sentence summary card

### Progressive Disclosure on Event Cards
- **Default:** 1-sentence summary (template)
- **On tap:** Expands to 3-sentence deeper explanation (LLM-generated, cached in memory per session). If enrichment not ready, show loading shimmer.
- **UI:** Small chevron/expand icon on each card

### Tyre Strategy Predictor
- **Model:** Rules-based engine (not LLM). Inputs: current lap, tyre age, compound, total laps, stops already made.
- **Logic:** Baseline compound life (soft: ~18, medium: ~30, hard: ~40). If remaining laps > remaining compound life, predict additional stop.
- **Confidence:** High (>80% compound life used), Medium (50-80%), Low (<50%)
- **Display:** "1-stop" / "2-stop" / "2+ stop" badge per driver with confidence dot (green/yellow/gray)
- **Known limitation:** Fixed values are averages. Circuit-specific calibration is Phase 2.

### Post-session "What You Learned" Card
- **Trigger:** Session status = "finished" from OpenF1. Covers race and qualifying.
- **Content:** Aggregates unique event types where a template was shown. Formats as "Today you learned about: [list]"
- **Storage:** localStorage/chrome.storage.local. Retained until next session starts.

### Race Mood/Excitement Indicator
- **Algorithm:** Rolling 5-event excitement score. Weights: safety car=4, overtake=3, gap collapse=2, pit stop=1, DRS=1, tyre degradation=1
- **Display:** Color intensity bar in sidebar header (cool blue = quiet, warm orange/red = exciting)
- **Updates:** Recalculated on every new event

### Mid-race Depth Toggle
- **Default:** Beginner mode (session-level). Toggle is in-session override.
- **Beginner:** Plain English, more frequent cards, jargon tooltips
- **Regular:** Denser info, fewer cards, no tooltips
- **UI:** Single toggle switch in sidebar header, persists for session duration only

### Replay Mode
- **Data source:** OpenF1 historical REST API. Full session pre-fetched on replay start.
- **Behavior:** `setInterval` at 1s ticks (1x speed default, 2x/4x configurable). Events fire when simulated clock passes their timestamp.
- **Time scrubber:** Scrubbing forward fires skipped events instantly (cards only). Scrubbing backward resets event feed.
- **No live features:** Catch-up button disabled. Mood indicator computed against simulated time.

## Voice Orchestration Rules

| Priority | Event Type | Notes |
|----------|-----------|-------|
| 1 (highest) | Safety car / VSC | Changes entire race |
| 2 | Pit stop | Strategy-defining |
| 3 | Overtake | Exciting, time-sensitive |
| 4 | Gap collapse / undercut | Strategic, can wait 5s |
| 5 | DRS activation | Contextual |
| 6 (lowest) | Tyre degradation | Advisory |

- Max 2 events in voice queue. If full, new events get card-only treatment.
- High-priority events interrupt current utterance.
- Voice reads the active template variant (beginner/regular) at fire time.
- Auto-dismiss after 10 seconds.
- Voice suppressed when toggle is Cards-only.

## Reviewer Concerns (from adversarial review, 3 rounds)

These were identified by independent AI review and should be resolved during engineering:

1. **Pre-race briefing data gap.** OpenF1 doesn't expose standings. Need Ergast API or scope briefing to grid + circuit only.
2. **chrome.storage.sync has 100KB quota.** Use chrome.storage.local for session data. Only prefs in sync.
3. **LLM call volume uncapped.** Consider ~30 calls/session cap with graceful degradation.
4. **Replay 2x/4x may be YAGNI.** 1x sufficient for demos. Consider deferring variable speed.
5. **Service worker keepalive.** If polling moves to SW, need chrome.alarms or keepalive strategy.
6. **Voice autoplay policy.** Chrome may block SpeechSynthesis without user gesture. Test early.
